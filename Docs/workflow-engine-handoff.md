# 工作流引擎（Workflow Engine）—— 跨仓库交接文档

> 状态：交接稿（handoff）
> 读者：在**独立 git 仓库 / 独立会话**中开发工作流引擎与可视化编辑器的人。
> 配套：QCode 侧设计见 `expert-system-design.md`（专家系统）。本文自包含，无需读 QCode 源码即可开工，但**第 4 章的对接契约必须与 QCode 侧保持一致**。

---

## 1. 目的（这个东西到底解决什么）

QCode（一个 Roo Code 的 VS Code 扩展 fork）要支持**长程任务**，任务分两类：

- **类型 A — 确定型**：流程已固化，按预定义工作流执行。**← 本工作流引擎服务于这一类。**
- **类型 B — 探索型**：无固定流程，LLM 自驱 loop（由 QCode 直接处理，不在本仓库范围）。

工作流引擎要达成三件事：

1. **可视化编辑**：用户能拖拽节点、连线、配参数，画出一条工作流。
2. **可执行**：把画好的图跑起来，每个节点驱动 QCode 去调工具 / 跑技能 / 派子专家 / 让 LLM 判断。
3. **可被复用为"技能/工具"**：一条工作流可以独立运行，也可以被一个探索型专家（类型 B）当成"一个高级动作"调用。

> 一句话：**让"已知怎么做"的流程，变成可视化编排、可执行、可被当技能调用的资产。**

---

## 2. 架构：两层，不要找大而全的现成产品

工作流系统拆成两个解耦的层。**不要引入 n8n / Node-RED / Temporal / LangGraph 这类重型产品**——它们是独立 server / 自带世界观，嵌进 VS Code 扩展是反模式，且它们的节点不认识 QCode 的"专家/技能/工具"。

| 层                 | 做法             | 选型                                                              |
| ------------------ | ---------------- | ----------------------------------------------------------------- |
| **① 可视化编辑器** | 节点式画布 UI    | **React Flow（`@xyflow/react`）**，MIT 协议，节点式编辑器事实标准 |
| **② 执行引擎**     | 把图按拓扑跑起来 | **自写轻量 DAG 解释器**（约几百行），节点动作 dispatch 给 QCode   |

为什么执行引擎要自己写：工作流的节点不是"发 HTTP 请求"这种通用动作，而是"调 QCode 的某个工具 / 跑某个技能 / 派一个子专家"——**这些动作只有 QCode 知道怎么执行**。所以引擎本身很薄：遍历图 → 对每个节点判断类型 → 调对应的 dispatch 入口（见第 4 章）。

## 3. 数据模型

### 3.1 图定义格式 = JSON

直接用 React Flow 的原生导入/导出格式（`nodes[] + edges[]`），不另造格式。一条工作流就是一份 JSON。

### 3.2 节点类型词汇表（★ 必须三方冻结后再开工 ★）

这是**编辑器、执行引擎、QCode 三方共同依赖的地基**。编辑器的"可拖拽节点种类"、引擎的"dispatch 分支"、QCode 的"执行入口"必须一一对应。开发前先冻结这张表：

| `type`      | 含义                     | 节点 `data` 需携带的关键字段（建议）        |
| ----------- | ------------------------ | ------------------------------------------- |
| `tool`      | 调用一个 QCode 工具      | `toolName`, `params`                        |
| `skill`     | 运行一个技能             | `skillName`, `args`                         |
| `expert`    | 派生子专家并等待其汇报   | `expertId/mode`, `subtaskPrompt`            |
| `llm`       | 让 LLM 做一次判断 / 生成 | `prompt`, `outputSchema?`                   |
| `condition` | 条件分支                 | `expression`（依据上游节点输出求值）        |
| `parallel`  | 并发执行多分支           | —（**先占位，QCode 并发能力就绪前不启用**） |

> 节点间数据传递：每个节点执行后产出一个结果对象，下游节点（尤其 `condition` / `llm`）可引用上游结果。建议约定一个简单的引用语法（如 `{{nodeId.output}}`），三方统一。

### 3.3 工作流元数据

每份工作流 JSON 顶层带：`name`、`description`、`version`、`inputs`（启动参数）、`nodes`、`edges`。`name`/`description` 用于注册成技能时的展示与触发。

## 4. 对接契约（与 QCode 的边界）★ 核心 ★

工作流引擎**不直接操作 QCode 内部**，只通过下面两个约定与 QCode 交互。

### 4.1 工作流即技能（Workflow-as-Skill）

- 一条工作流 JSON **注册为 QCode `.roo/skills` 下的一个技能**（数据驱动，无需改 QCode 代码）。
- 注册后两种用法：
    - **独立运行**：作为类型 A 专家的执行体。
    - **被调用**：类型 B 探索型专家在自驱 loop 中把它当一个工具调用（例："走一遍 `release-flow`"）。

### 4.2 节点执行入口（Node Dispatch Interface）

QCode 需要（由 QCode 侧实现并暴露）一个稳定入口：**给定一个节点（type + data）+ 当前上下文，执行它并返回结果**。执行引擎只负责"遍历图、决定下一个跑哪个节点、把结果在节点间传递"，**具体每个节点怎么落地由 QCode 完成**：

```
dispatchNode(node, context) -> nodeResult
  node.type === 'tool'      → 调 QCode 工具
  node.type === 'skill'     → 跑 QCode 技能
  node.type === 'expert'    → new_task 派子专家，等其摘要汇报
  node.type === 'llm'       → 走一次 Task loop 的 LLM 调用
  node.type === 'condition' → 求值，决定走哪条 edge
```

> **职责边界**：引擎 = 控制流（图遍历 / 分支 / 数据传递）；QCode = 动作落地（tool/skill/task/llm 的真正执行）。这条线要清晰，避免引擎里塞 QCode 专属逻辑。

### 4.3 约束（来自 QCode，必须遵守）

- **禁止引入 AI-SDK / `@ai-sdk/*` / `ai` 依赖**（QCode 刻意不用）。
- **第一阶段不做并发**：`expert`/`parallel` 节点先按**串行**实现（QCode 当前是单活动任务模型，并发需后续重构）。`parallel` 节点可在编辑器里占位，但执行先降级为串行或禁用。
- **子专家只回传"结论摘要"**，不回传完整对话历史（防止上下文爆炸）。

---

## 5. 建议的里程碑

1. **冻结节点类型词汇表（3.2）+ 图 JSON schema（3.3）** —— 三方对齐，先于一切编码。
2. **可视化编辑器 MVP**：React Flow 画布 + 上述节点类型 + 导出/导入 JSON。
3. **执行引擎 MVP**：读 JSON、拓扑遍历、串行执行、节点间数据传递、`condition` 分支。先用 mock 的 `dispatchNode` 跑通控制流。
4. **与 QCode 对接**：把 mock 的 `dispatchNode` 换成真实入口（4.2），工作流注册为 skill（4.1）。
5. **闭环验证**：画一条简单工作流（如 read → llm 判断 → tool）→ 注册为技能 → 由一个类型 A 专家跑通。
6. （后续）并发节点、子专家并行 —— 待 QCode 重构单活动任务模型后。

---

## 6. 两个会话的协同点（重要）

- **节点类型词汇表（3.2）** 是 QCode 会话与本会话的**共享契约**，任一方改动都要同步。建议把这张表作为唯一事实来源（single source of truth），两边都引用它。
- QCode 侧第一阶段交付：**统一专家抽象 + 类型 B 自驱 loop + `dispatchNode` 入口（4.2）+ 串行子专家**。本仓库可在 `dispatchNode` 就绪前用 mock 先行开发到里程碑 3。
